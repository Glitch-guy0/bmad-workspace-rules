# Repository & Agent File Structure

Two-repo architecture, DTO handoffs, file trees, agent lifecycle, stress test, naming conventions, and the complete cross-repo document map.

## Repo Architecture

- Two repos: Product Repo (owned by Product Agent + single user, purpose: capture → validate → confirm) and Developer Repo (owned by Senior Developer Agent, purpose: plan → implement → report)
- Product repo generates Milestone Confirmation File (DTO OUT) and receives Implementation Report (DTO IN)
- Dev repo receives Milestone Confirmation File (DTO IN) and generates Implementation Report (DTO OUT)
- Boundary enforced by repos. Product repo never contains implementation code. Developer repo never contains unvalidated ideas. Only DTO files cross the boundary

## DTO — File Handoffs

### DTO 1 — Milestone Confirmation File

- Path: product-repo/dto/outgoing/[milestone-slug]-confirmed.md → dev-repo/dto/incoming/[milestone-slug]-confirmed.md
- Trigger: User stress test passed + PM + engineering lead sign-off complete. File name: `[milestone-slug]-confirmed.md`
- Read-only after confirmation. Dev agent starts nothing until this file exists

### DTO 2 — Implementation Report

- Path: dev-repo/dto/outgoing/[milestone-slug]-report.md → product-repo/dto/incoming/[milestone-slug]-report.md
- Trigger: Implementation complete, all DoD items checked. File name: `[milestone-slug]-report.md`
- Template: Milestone name, slug, Report version (1.0), Status (Complete/Partial/Blocked), Generated by (Developer Agent), Date, Linked DTO in, What shipped (per requirement with ✅/⚠️/❌), Deviations from confirmed scope, New decisions made (ADRs filed), Product doc updates required (current-state-map, feature-index, evolution-timeline, direction), Evolution timeline entry (ready to paste), Feature index entries (ready to paste), Retro note

### DTO 3 — NFR Proposal (dev→product)

- Documented in 03-nfr-tech-debt-indexing.md

## Product Repo File Structure

```
product-repo/
  README.md, AGENT-INSTRUCTIONS.md
  dto/incoming/ [slug]-report.md
  dto/outgoing/ [slug]-confirmed.md
  product/ product-brief.md, evolution-timeline.md, current-state-map.md, direction.md
  feature-index/ _index.md, [feature-slug].md
  ideas/ _index.md, [idea-slug].md
  milestones/ _index.md
    active/[slug]/ milestone.md, requirements/[req-slug].md, stress-test/session-[date].md
    confirmed/[slug]/ milestone-confirmed.md, sign-off-log.md
    completed/[slug]/ milestone-confirmed.md, sign-off-log.md, retro-note.md
  reports/ [slug]-report.md
```

Key rules: ideas never enter milestones/active/ until validated; milestones/confirmed/ is read-only; dto/outgoing/ never edited after sending; current-state-map and feature-index updated only from incoming DTOs; stress test sessions created fresh each time.

## Developer Repo File Structure

```
dev-repo/
  README.md, AGENT-INSTRUCTIONS.md
  dto/incoming/ [slug]-confirmed.md
  dto/outgoing/ [slug]-report.md
  planning/[slug]/ tdd.md, erd.md, adrs/ADR-[n]-[title].md, api-contracts/[endpoint].md, spikes/[slug].md, implementation-brainstorm.md, atomic-change-plan.md, test-scenarios.md
  implementation/[slug]/ progress.md, decisions-log.md
  reports/ [slug]-report.md
  archive/[slug]/ [all files]
```

Key rules: no planning file until dto/incoming/ has confirmed milestone; dto/incoming/ never edited; ADRs filed at moment of decision; decisions-log promoted to ADRs before report; archive append-only; report not generated until all DoD items in atomic-change-plan checked.

## Agent Flow

### Product Agent (6 phases)

1. Idea: user submits idea → creates ideas/[slug].md, updates ideas/_index.md
2. Validation: explore idea; if validated → create milestones/active/[slug]/milestone.md, requirements, update milestones/_index.md
3. Stress test: structured Q&A with user to pressure-test requirements → session log; repeat until all open questions resolved, AC complete, scope locked
4. Sign-off: present final milestone for approval → create confirmed milestone, sign-off-log, copy to dto/outgoing
5. Waiting: monitor dto/incoming/ for implementation report
6. Update (triggered by incoming report): update current-state-map, evolution-timeline, feature-index, move milestone to completed, append retro note, archive report

### Developer Agent (6 phases)

1. Receipt: detect dto/incoming/[slug]-confirmed.md → read fully, create planning/[slug]/ directory
2. Technical design: create TDD, ERD (if schema changes), API contracts, spikes, ADRs
3. Validation (Gate 2): self-check TDD against confirmed milestone. Flag gaps for human review
4. Implementation planning: create brainstorm, test scenarios, atomic change plan, progress log
5. Implementation: update progress per atomic change, update decisions-log, create new ADRs, check off items in atomic change plan
6. Report: promote decisions-log → remaining ADRs, generate report, copy to dto/outgoing, archive all files

## Stress Test

- Validation gate between draft milestone and confirmed one. Run by product agent with single user
- Per requirement: clarity check (real scenario, success definition), edge case pressure (failure conditions), scope boundary test (adjacent features, existing users), conflict detection (current state map contradictions), dependency reveal (preconditions)
- Session log format: Date, Session number, Agent, User, Requirements reviewed, Findings per requirement (issue/resolution/AC updated), Open items (owner, must resolve before sign-off), Next session needed
- Sign-off blocked until all open items across all sessions resolved

## File Naming Conventions

- Slugs: kebab-case (milestone: multi-photo-upload, idea: bulk-export-feature, nfr: nfr-upload-concurrency, tech-debt: debt-session-management, feature: photo-gallery)
- Dates: ISO 8601 YYYY-MM-DD. Session files: stress-test/session-2026-05-21.md
- ADR numbering: padded integers, sequential per repo (ADR-001, ADR-002). Never reuse number. Superseded ADR links to replacement
- Versioning: inside confirmed file header (v1.0, v1.1, v2.0). Never in filename. Version increments logged in sign-off-log.md
- Status: never in filename. Lives in file header. Lifecycle via folder location (active → confirmed → completed; postponed → nfr-proposals → planning)
